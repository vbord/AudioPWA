[app.js : line ~490]
player.currentTime = position;
COMMENT:
- This is a RESUME seek.
- Can happen when restoring playback after app reload or navigation.
- Can fire more than once if:
  - resume logic is triggered again after UI expansion
  - or playFile() is indirectly re-entered (e.g. target.click())
- High risk for duplicate seek during one resume cycle.

[app.js : line ~373]
player.currentTime = userProgress[book].position;
COMMENT:
- This is ALSO a RESUME seek.
- Usually occurs inside playFile() when hasSaved === true.
- Can happen in the SAME playback session as the line above.
- If both execute, browser receives TWO seeks to same position.
- This is the PRIMARY cause of slow seek near end.
